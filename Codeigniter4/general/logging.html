로그 수준

log_message() 메서드를 사용하여 로컬 로그 파일에 정보를 기록할 수 있습니다. 첫 번째 매개변수에 오류의 “수준”을 제공하여 메시지 유형(디버그, 오류 등)을 표시해야 합니다. 두 번째 매개변수는 메시지 자체입니다.

<?php

if ($some_var === '') {
    log_message('error', 'Some variable did not contain a value.');
}

RFC 5424 수준과 일치하는 8가지 로그 수준이 있으며 그 수준은 다음과 같습니다.

수준

설명

debug

상세 디버그 정보.

info

사용자 로그인, SQL 쿼리 로깅 등과 같은 애플리케이션의 흥미로운 이벤트

notice

애플리케이션에서 일반적이지만 중요한 이벤트입니다.

warning

더 이상 사용되지 않는 API 사용, API의 잘못된 사용 또는 반드시 잘못된 것은 아니지만 기타 바람직하지 않은 일과 같이 오류가 아닌 예외적인 발생입니다.

error

즉각적인 조치가 필요하지 않지만 일반적으로 기록하고 모니터링해야 하는 런타임 오류입니다.

critical

애플리케이션 구성요소를 사용할 수 없거나 예상치 못한 예외와 같은 심각한 상황입니다.

alert

전체 웹사이트가 다운되거나 데이터베이스를 사용할 수 없는 경우 등 즉시 조치를 취해야 합니다.

emergency

시스템을 사용할 수 없습니다.

로깅 시스템은 시스템 관리자나 웹마스터에게 이러한 이벤트에 대해 경고하는 방법을 제공하지 않으며 정보만 기록합니다. 더 중요한 이벤트 수준의 경우 위에서 설명한 오류 처리기에 의해 자동으로 로깅이 발생합니다.

설정

app/Config/Logger.php 설정 파일 내에서 실제로 기록되는 수준을 수정할 수 있을 뿐만 아니라 다양한 수준을 처리하기 위해 다양한 로거를 할당할 수도 있습니다.

설정 파일의 threshold 값은 애플리케이션 전체에 기록되는 수준을 결정합니다. 애플리케이션에서 기록하도록 요청한 수준이 있지만 임계값으로 인해 현재 기록이 허용되지 않는 경우 해당 수준은 무시됩니다. 가장 간단한 사용 방법은 이 값을 기록하려는 최소 수준으로 설정하는 것입니다. 예를 들어 정보 메시지가 아닌 경고 메시지를 기록하려면 임계값을 5로 설정합니다. 수준 5 이하(런타임 오류, 시스템 오류 등 포함)의 모든 로그 요청이 기록되고 정보, 알림 및 디버그는 무시됩니다.

<?php

namespace Config;

use CodeIgniter\Config\BaseConfig;

class Logger extends BaseConfig
{
    public $threshold = 5;

    // ...
}

수준의 전체 목록과 해당 임계값은 참조용 설정 파일에 있습니다.

로그 수준 번호 배열을 임계값에 할당하여 기록하려는 특정 수준을 선택하고 선택할 수 있습니다.

<?php

namespace Config;

use CodeIgniter\Config\BaseConfig;

class Logger extends BaseConfig
{
    // Log only debug and info type messages
    public $threshold = [5, 8];

    // ...
}

여러 로그 처리기 사용

로깅 시스템은 동시에 실행되는 로깅을 처리하는 여러 방법을 지원할 수 있습니다. 각 핸들러는 특정 수준을 처리하고 나머지는 무시하도록 설정할 수 있습니다. 현재 세 개의 핸들러가 기본 설치와 함께 제공됩니다.

  • File Handler은 기본 처리기이며 매일 로컬에서 단일 파일을 생성합니다. 이는 권장되는 로깅 방법입니다.

  • ChromeLogger Handler Chrome 웹 브라우저에 ChromeLogger extension이 설치되어 있는 경우 이 핸들러를 사용하여 Chrome 콘솔 창에 로그 정보를 표시할 수 있습니다.

  • Errorlog Handler 이 핸들러는 PHP 기본 error_log() 함수를 활용하고 거기에 로그를 작성합니다. 현재 error_log()의 0 및 4 메시지 유형만 지원됩니다.

핸들러는 기본 설정 파일의 $handlers 속성에서 구성됩니다. 이는 단순히 핸들러 및 해당 구성의 배열입니다. 각 핸들러는 완전한 네임스페이스 클래스 이름인 키로 지정됩니다. 값은 각 핸들러에 특정한 다양한 속성의 배열입니다. 각 핸들러의 섹션에는 하나의 공통 속성인 handles이 있습니다. 이는 핸들러가 정보를 기록할 로그 수준 이름의 배열입니다.

<?php

namespace Config;

use CodeIgniter\Config\BaseConfig;

class Logger extends BaseConfig
{
    public $handlers = [
        // File Handler
        'CodeIgniter\Log\Handlers\FileHandler' => [
            'handles' => ['critical', 'alert', 'emergency', 'debug', 'error', 'info', 'notice', 'warning'],
        ],
    ];

    // ...
}

컨텍스트로 메시지 수정

기록되는 이벤트의 컨텍스트에 따라 메시지의 세부 정보를 수정하려는 경우가 많습니다. 사용자 ID, IP 주소, 현재 POST 변수 등을 기록해야 할 수도 있습니다. 메시지에 자리 표시자를 사용하면 됩니다. 각 자리 표시자는 중괄호로 묶어야 합니다. 세 번째 매개변수에는 자리 표시자 이름(중괄호 제외)과 해당 값의 배열을 제공해야 합니다. 메시지 문자열에 다음이 삽입됩니다.

<?php

// Generates a message like: User 123 logged into the system from 127.0.0.1
$info = [
    'id'         => $user->id,
    'ip_address' => $this->request->getIPAddress(),
];

log_message('info', 'User {id} logged into the system from {ip_address}', $info);

예외 또는 오류를 기록하려면 ‘예외’ 키를 사용하고 값은 예외 또는 오류 자체를 사용할 수 있습니다. 오류 메시지, 파일 이름 및 줄 번호를 포함하는 해당 개체에서 문자열이 생성됩니다. 메시지에 예외 자리 표시자를 제공해야 합니다.

<?php

try {
    // Something throws error here
} catch (\Exception $e) {
    log_message('error', '[ERROR] {exception}', ['exception' => $e]);
}

현재 페이지 요청에 따라 자동으로 확장되는 몇 가지 핵심 자리 표시자가 있습니다.

플레이스홀더

삽입 값

{post_vars}

$_POST 변수

{get_vars}

$_GET 변수

{session_vars}

$_SESSION 변수

{env}

현재 환경 이름(예: development)

{file}

로거를 호출하는 파일 이름

{line}

로거가 호출된 {file}의 줄

{env:foo}

$_ENV의 ‘foo’ 값

서드파티 로거 사용

Psr\Log\LoggerInterface에서 확장되고 PSR-3과 호환되는 한 원하는 다른 로거를 사용할 수 있습니다. 즉, PSR-3 호환 로거를 쉽게 사용하거나 직접 만들 수 있습니다.

app/Config/Autoload.php 설정 파일에 추가하거나 Composer와 같은 다른 오토로더를 통해 시스템에서 타사 로거를 찾을 수 있는지 확인해야 합니다. 다음으로 app/Config/Services.php을 수정하여 logger 별칭이 새 클래스 이름을 가리키도록 해야 합니다.

이제 log_message() 함수를 통해 수행되는 모든 호출은 대신 라이브러리를 사용합니다.
